ADR-008 — Cola de comandos offline cliente PWA

Cómo el técnico graba trabajo en campo sin conectividad: el cliente almacena comandos (intenciones) en IndexedDB, no eventos. El servidor decide y produce eventos en Marten al sincronizar.

3. Ciclo de vida de un comando

Cada comando en comandos_pendientes avanza por estos estados. Las transiciones rojas son terminales que requieren acción del técnico.

4. Flujo de adjuntos en 2 fases

Las fotos no viajan en el payload del comando. Patrón SAS de ADR-005: el server emite URI firmada, el cliente sube directo a Blob, después confirma. Esto soporta fotos de 5-10 MB sin saturar el body de POST y sin pasar por backend.

💡 Por qué 2 fases: si embebiéramos la foto en el comando, un POST de 50 MB con red intermitente fallaría completo y perderíamos el comando. Separando: el comando viaja liviano, la foto va directo a Blob (que tolera reintentos parciales), y la confirmación es atómica.

5. Schema en cliente — IndexedDB

Tres object stores. Sólo comandos_pendientes sincroniza al server; los otros dos son locales.

Store: comandos_pendientes

Cola FIFO por streamId. Sólo se sincroniza este store al server. Indices: (streamId, capturadoEn) para FIFO por stream, status para el worker, (status, capturadoEn) para reintentos.

CampoTipoNotas
clientCommandIdUUIDv7PK. Idempotency key, viaja como X-Client-Command-Id
tipostringRegistrarHallazgo, FirmarInspeccion, RegistrarPreop, ...
streamIdGuidInspeccionId / SeguimientoHallazgoId / NovedadPreopId. Index para FIFO por stream
payloadobjectDatos del comando (ej. para RegistrarHallazgo: parteId, causaFallaId, descripcion, ...)
capturadoEnISO timestamp clienteReconstruye orden causal — el server lo persiste tal cual
ubicacionGpsUbicacionGpsCapturada al momento del comando. V-F3 requiere GPS al firmar
claimsSnapshotobject{ tecnicoId, capabilities[] } al momento de captura
statusenumpendiente | enviando | confirmado | rechazado

ultimoErrorstring?Razón del último 4xx/5xx (V-F#, I-#, mensaje del server)

Store: catalogos

Read-only desde ERP por sync nocturno (ADR-004). Stale-while-revalidate hasta 24h. No sincroniza al server.

CampoNotas
tipoequipos | rutinas | partes | actividades | causas | tipos-falla | repuestos | ubicaciones | obras

Store: read_models

Vista optimista del trabajo en curso. Derivada — se reemplaza con la versión autoritativa del server cuando llega vía SignalR push o tras ack. No sincroniza.

CampoNotas

stateEstado proyectado tras aplicar comandos pendientes sobre snapshot del server
baseVersionVersión del agregado server tomada como base. Si avanza por SignalR, se rebaseo
tieneCambiosPendientesBool — para mostrar badge "no sincronizado" en lista de inspecciones

6. Anatomía de una jornada — estructura física en cliente

Para que el modelo "comandos en cola" sea concreto, esta sección muestra cómo se llena el storage local durante una jornada típica del técnico, hito por hito.

6.1 Definición Dexie (TypeScript)

Una sola declaración al arrancar la app define los tres object stores y sus índices:

import Dexie, { Table } from 'dexie';

interface ComandoPendiente {
  clientCommandId: string;       // UUIDv7 — PK
  tipo: string;                  // "RegistrarHallazgo", "FirmarInspeccion", ...
  streamId: string;              // InspeccionId / SeguimientoHallazgoId
  payload: Record<string, unknown>;
  capturadoEn: string;           // ISO timestamp cliente
  ubicacionGps: { lat: number; lon: number; precisionMetros: number; capturadoEn: string };
  claimsSnapshot: { tecnicoId: number; capabilities: string[] };
  status: 'pendiente' | 'enviando' | 'confirmado' | 'rechazado';
  intentos: number;
  ultimoError?: string;
}

class InspeccionesDB extends Dexie {
  comandos_pendientes!: Table<ComandoPendiente, string>;
  catalogos!: Table<CatalogoCache, string>;
  read_models!: Table<ReadModel, string>;

  constructor() {
    super('inspecciones_offline');
    this.version(1).stores({
      comandos_pendientes:
        'clientCommandId, streamId, status, [streamId+capturadoEn], [status+capturadoEn]',
      catalogos: 'tipo, lastSyncedAt',
      read_models: 'streamId, tipo, tieneCambiosPendientes',
    });
  }
}

export const db = new InspeccionesDB();

[streamId+capturadoEn] es índice compuesto que habilita lectura FIFO por stream para el sync worker.

6.2 OPFS para fotos (no en IndexedDB)

Las fotos van al Origin Private File System vía navigator.storage.getDirectory(). Ineficiente meterlas en IndexedDB cuando hay binarios pesados (5-10 MB c/u).

/adjuntos_pendientes/
  ├── {adjuntoTempId-1}.jpg     (~3 MB)
  ├── {adjuntoTempId-2}.jpg
  └── {adjuntoTempId-3}.jpg
/adjuntos_subiendo/
  └── {adjuntoTempId-N}.jpg     (movido cuando arranca PUT a Blob)

Cuando el upload SAS confirma (AdjuntoConfirmado_v1 recibido), el archivo se elimina del OPFS.

6.5 Persistencia entre cierres de la PWA

IndexedDB y OPFS persisten en disco, no son memoria de sesión. Si el técnico cierra la pestaña, reinicia el celular, o pasa una semana sin abrir la app — al reabrir, todo está como lo dejó (comandos en cola con su intentos/ultimoError, fotos en OPFS, read_models intactos).

Lo único que no persiste son las variables JS en memoria (estado de React, etc.). Por eso el bootstrap al arrancar la app hace 4 pasos:

async function bootstrap() {
  await db.open();                                  // 1. reconectar a IndexedDB

  const inspeccionesActivas = await db.read_models  // 2. reanimar UI desde
    .where('tipo').equals('Inspeccion')             //    proyecciones locales
    .toArray();                                     //    (no llama al server todavía)
  setUIFromReadModels(inspeccionesActivas);

  startSyncWorker();                                // 3. levantar worker; si hay red,
                                                    //    empieza a vaciar la cola
  refreshCatalogosIfStale();                        // 4. en paralelo, refrescar
                                                    //    catálogos si pasaron >24h
}

El storage local se pierde solo si: (a) el usuario limpia datos del browser, (b) el browser hace eviction por presión de cuota agresiva, (c) el usuario desinstala la PWA. Caso (b) lo mitigamos monitoreando navigator.storage.estimate() y bloqueando suavemente al técnico al llegar a <20 % libre.

6.7 Comportamiento por plataforma — iOS vs Android

PWA es un estándar web, pero el comportamiento real del storage local difiere significativamente entre iOS Safari y Android Chrome/Edge. Esto afecta decisiones operativas del rollout.

Aspecto 🤖 Android (Chrome/Edge/Samsung) 🍎 iOS Safari (iPhone/iPad)
IndexedDB✅ Robusto, persistencia confiable✅ Funciona desde iOS 17.4 (marzo 2024)
OPFS✅ Completo (Chrome 102+)✅ Soportado desde iOS 16+
Cuota típica~60 % del disco libre (típicamente GB)~1 GB nominal, expandible con persist() sin garantía
Background Sync API✅ Funciona en background❌ NO existe en iOS Safari
Periodic Background Sync✅ Chrome/Edge❌ NO existe
Web Push✅ Siempre✅ Solo si está instalada (Add to Home Screen), iOS 16.4+
Service Worker✅ Robusto⚠️ Funciona pero con bugs históricos
storage.persist()✅ Suele concederse⚠️ Suele rechazarse
Eviction por inactividad (ITP)❌ No aplica⚠️ 7 días sin abrir → wipe del storage
⚠️ Riesgo crítico iOS — regla de 7 días. Si el técnico no abre la PWA por 7 días consecutivos, Safari borra automáticamente todo el storage de origen (IndexedDB, OPFS, localStorage, cookies). Es política deliberada de Apple (Intelligent Tracking Prevention), no bug. Si el técnico se va de vacaciones 10 días con comandos pendientes en cola, el trabajo se pierde sin recuperación cliente-side. PWA instalada (Add to Home Screen) tiene reloj ITP separado y algo más permisivo.
iOS sin Background Sync. La cola solo se vacía con app en foreground. Si el técnico cierra la app antes de que vuelva la red, la cola espera al próximo abrir manual.
Cuotas más conservadoras en iOS. Una jornada de 90 MB cabe; un técnico que arrastra 3-4 jornadas sin sincronizar puede topar antes que en Android.

Mitigaciones de código

async function ensurePersistence() {
  // 1. Pedir persistencia explícita (Android la concede, iOS a veces)
  const granted = await navigator.storage.persist();

  // 2. Detectar plataforma y mostrar warnings dirigidos
  const isIOS = /iPad|iPhone|iPod/.test(navigator.userAgent);
  const isStandalone = window.matchMedia('(display-mode: standalone)').matches;

  if (isIOS && !isStandalone) {
    showBanner('Para trabajar offline, instala la app: ' +
               'Compartir → Añadir a pantalla de inicio');
  }

  // 3. Métricas de salud de cola — alertar si se acerca al riesgo iOS
  const oldestPending = await db.comandos_pendientes
    .where('status').equals('pendiente')
    .first();
  if (oldestPending && diasSinSincronizar(oldestPending) > 4) {
    showWarning(
      `Hay trabajo sin sincronizar hace ${dias} días. ` +
      `iOS borra el storage tras 7 días sin abrir la app.`
    );
  }
}

Acciones operativas (vinculantes para el rollout)

  1. Requisito de instalación PWA en runbook del técnico — la app debe estar agregada a pantalla de inicio antes de salir a campo.
  2. Heartbeat diario vía push notification (iOS 16.4+ soporta push en PWAs instaladas) — abre la app brevemente y dispara sync, manteniendo el reloj ITP reseteado.
  3. Alerta de cola estancada a partir del día 4 sin sync (banner + push si la app no se abre).
  4. Métrica server-side "técnicos con datos no sincronizados >5 días" → backoffice los contacta proactivamente.
  5. Decisión default: aceptar iOS con mitigaciones; reevaluar si emerge pérdida real de datos en operación.

Camino de evolución si iOS resulta inaceptable

  • Capacitor / PWA-wrapper nativo iOS — la app sigue siendo React/MUI pero compilada como app nativa con storage nativo (Core Data / SQLite) sin ITP. Costo: ~2-3 semanas + Apple Developer ($99/año) + distribución TestFlight/App Store. La lógica del ADR-008 (cola, sync, idempotencia) se reutiliza tal cual.
  • Restringir iOS a uso solo-online y obligar Android para campo offline. Requiere alineación con cliente piloto.
✅ Lo bueno: los técnicos con Android tienen experiencia offline robusta sin caveats. El módulo está diseñado para que la lógica funcione idéntica en ambas plataformas; lo que cambia son los riesgos de pérdida y las métricas de monitoreo.

7.13 Patrones de mitigación transversales

Cinco mecanismos cubren la mayoría de los modos de falla anteriores:

  1. Idempotencia end-to-end — clientCommandId UUIDv7 → Wolverine MessageId → claves de negocio en handlers (HallazgoId, AdjuntoId). Defiende contra retries, dedup expirado y crashes parciales.
  2. Atomicidad transaccional — un solo SaveChangesAsync commitea eventos + envelope + outbox + proyecciones inline. Regla CLAUDE.md no negociable.
  3. Stop-on-error por stream — aísla problemas a un stream sin bloquear la jornada del técnico ni otros streams.
  4. Pull al bootstrap + push en vivo — SignalR no es source of truth, solo optimización. Catch-up vía REST garantiza eventual consistency aunque se pierdan pushes.
  5. Defensa en profundidad — cada capa asume que la anterior puede fallar: cliente reintenta, server deduplica, handlers validan claves de negocio, blob hace HEAD antes de confirmar, cron limpia huérfanos.

7.14 Gaps documentados sin mitigación al 100 %

Estos gaps son aceptados conscientemente para v1.0 y tienen camino de evolución claro:

iOS ITP 7 días. Mitigaciones reducen incidencia pero no eliminan riesgo de pérdida de cola. Camino: Capacitor wrapper iOS.
Jornadas offline >24h. Refresh token expira; trabajo bloqueado hasta re-login online. Camino: claimsSnapshot server-side (opción b del manejo de auth offline).
Foto perdida en OPFS antes de subir. No recuperable. Mitigación parcial: UI muestra adjunto pendiente y técnico puede volver a tomar foto.
MYE no respeta Idempotency-Key. Riesgo de duplicado en ERP si MYE no implementa correctamente ADR-003. Fuera de control del módulo — requiere disciplina del equipo MYE.

8. Decisiones críticas y trade-offs

Por qué comandos y no eventos en cliente

  • Las invariantes del agregado (V-F1..V-F8, I-1..I-11) requieren validación contra el estado actual del stream — el cliente no es autoritativo de ese estado.
  • Multi-técnico en la misma inspección (§15.2 modelo): dos técnicos offline emitiendo eventos paralelos rompen la causalidad. Comandos serializados por el server resuelven esto.
  • Replay: si el cliente generó HallazgoRegistrado_v1 y el server espera _v2, el rebuild rompe. Comandos son contrato versionable más estable.

Por qué stop-on-error por stream (no global, no continue)

  • Global stop: un rechazo en la inspección X bloquearía la inspección Y de otro equipo — blast radius excesivo.
  • Continue tras error: si RegistrarHallazgo2 falla y seguimos con FirmarInspeccion, el firmar también falla porque el hallazgo no entró → cascada de rechazos confusos.
  • Stop por stream aísla el problema y le da al técnico la oportunidad de corregir solo lo que falló.

Trade-offs aceptados

iOS Safari sin Background Sync API. En iPad/iPhone el sync solo ocurre con app abierta. Si los técnicos usan iOS, deben volver a la app (o mantenerla en foreground) para vaciar la cola. Reevaluar en operación real con feedback de campo.
Refresh token TTL=24h limita ventana offline real. Jornadas >24h sin red requieren re-auth. v1.1 podría promover a claimsSnapshot server-side (requiere ampliar ADR-002).
Conflictos requieren intervención humana. No hay merge automático tipo CRDT. Aceptable porque el dominio (firmar inspección, generar OT) no es merge-friendly.
Cuota IndexedDB / OPFS. Una jornada larga: 100 fotos × 5 MB ≈ 500 MB. Cabe holgado en cuota típica (~60% del disco libre), pero monitoreamos con navigator.storage.estimate() y bloqueo suave al técnico al llegar a <20% libre.
